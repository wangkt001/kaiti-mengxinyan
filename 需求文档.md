# 校园二手交易系统需求文档

## 1. 项目背景与目标

### 1.1 项目背景

传统校园二手交易存在信息分散、信任缺失、管理无序等痛点，导致闲置资源流通效率低下，交易风险较高。随着数字化校园建设的推进，亟需一个规范化、便捷化的二手交易平台，以促进校园闲置资源的高效流通，营造绿色校园环境。

### 1.2 项目目标

- 搭建适配师生需求的数字化交易平台，实现闲置资源高效流通与交易规范管理
- 助力绿色校园、智慧校园建设，减少资源浪费
- 探索中小型Web应用的工程化开发路径，验证技术与校园场景的融合适配性
- 形成可复用开发范式，为校园数字化服务积累实践经验

## 2. 用户角色

| 角色   | 注册方式 | 核心权限                                   |
| ------ | -------- | ------------------------------------------ |
| 学生   | 学号注册 | 发布商品、购买商品、评价交易、查看个人信息 |
| 教师   | 工号注册 | 发布商品、购买商品、评价交易、查看个人信息 |
| 管理员 | 系统分配 | 管理用户、管理商品、处理纠纷、系统设置     |

## 3. 功能需求

### 3.1 用户管理模块

- **用户注册**：支持学生（学号）和教师（工号）注册，需验证身份信息
- **用户登录**：支持账号密码登录，使用Session认证
- **个人信息管理**：查看和修改个人基本信息，包括头像、联系方式等
- **密码修改**：支持用户修改登录密码
- **用户注销**：支持用户注销账号

### 3.2 商品管理模块

- **商品发布**：用户可发布二手商品，包括商品名称、描述、价格、图片、分类等信息
- **商品编辑**：发布者可编辑已发布商品的信息
- **商品删除**：发布者可删除已发布商品
- **商品列表**：支持按分类、价格、发布时间等条件筛选和排序
- **商品详情**：查看商品详细信息，包括图片、描述、价格、发布者信息等

### 3.3 交易管理模块

- **商品购买**：用户可购买商品，生成订单
- **订单管理**：查看个人订单列表，包括待支付、待发货、待收货、已完成、已取消等状态
- **交易状态更新**：卖家可更新订单状态（发货、完成），买家可更新订单状态（收货）
- **交易记录**：查看历史交易记录

### 3.4 评价管理模块

- **交易评价**：交易完成后，买卖双方可互相评价
- **评价查看**：查看用户的历史评价记录
- **评价统计**：显示用户的总体评价等级和评分

### 3.5 纠纷处理模块

- **纠纷提交**：用户可针对订单提交纠纷，填写问题说明
- **纠纷处理**：管理员查看纠纷内容并给出处理结果
- **纠纷记录**：用户可查看纠纷状态和处理结果

### 3.6 系统管理模块

- **用户管理**：管理员可查看、编辑、禁用用户账号
- **商品管理**：管理员可查看、编辑、下架违规商品
- **订单管理**：管理员可查看所有订单记录
- **纠纷管理**：管理员可处理用户提交的纠纷
- **系统设置**：管理员可设置系统参数，如商品分类、交易规则等

## 4. 非功能需求

### 4.1 性能需求

- 系统响应时间：页面加载时间不超过2秒
- 并发处理能力：支持同时在线用户数不低于1000人
- 数据库查询性能：复杂查询响应时间不超过1秒

### 4.2 安全需求

- 用户认证：使用Session认证，确保用户身份安全
- 数据加密：敏感信息（如密码）进行加密存储
- 权限控制：严格的角色权限管理，防止越权操作
- 输入验证：对用户输入进行严格验证，防止SQL注入等攻击
- 安全测试：通过漏洞扫描确认无高危漏洞

### 4.3 易用性需求

- 界面友好：采用直观、简洁的界面设计
- 响应式适配：支持PC端和移动端访问
- 操作便捷：简化操作流程，减少用户操作步骤
- 错误提示：提供清晰、友好的错误提示信息

### 4.4 可维护性需求

- 代码规范：遵循统一的代码规范和命名约定
- 模块化设计：采用模块化设计，便于代码维护和扩展
- 文档完善：提供详细的系统文档和开发文档

## 5. 界面简要描述

### 5.1 整体布局

- 顶部导航栏：包含logo、搜索框、用户中心、购物车等

## 4. 交互流程

1. **用户注册/登录流程**：
   - 新用户点击注册按钮 → 填写注册信息 → 验证身份 → 注册成功 → 自动登录
   - 已有用户点击登录按钮 → 填写登录信息 → 验证身份 → 登录成功 → 进入首页

2. **商品发布流程**：
   - 用户登录 → 点击发布商品 → 填写商品信息 → 上传图片 → 提交审核 → 发布成功

3. **商品购买流程**：
   - 用户浏览商品 → 点击商品详情 → 查看商品信息 → 点击购买 → 确认订单 → 支付 → 订单生成

4. **交易评价流程**：
   - 交易完成 → 系统提示评价 → 买卖双方互相评价 → 评价成功 → 显示评价记录

5. **纠纷处理流程**：
   - 用户提交纠纷 → 管理员处理并反馈结果 → 用户查看处理结果

## 3. 数据库设计说明

1. **用户表**：存储用户基本信息，区分学生、教师和管理员角色，使用学号/工号作为唯一标识。
2. **商品分类表**：存储商品分类信息，方便用户筛选商品。
3. **商品表**：存储商品详细信息，包括名称、描述、价格、库存等，关联到发布者和分类。
4. **商品图片表**：存储商品图片信息，一个商品可以有多个图片。
5. **订单表**：存储订单信息，包括买家、卖家、商品、价格、状态等。
6. **评价表**：存储交易评价信息，包括评价者、被评价者、评分、评价内容等。
7. **纠纷表**：存储交易纠纷信息，包括提交用户、订单、凭证材料、处理结果和关闭时间等，用于支撑纠纷处理闭环。

## 5. 系统架构

### 5.1 前端架构

- 采用Vue 3组合式API，实现组件化开发
- 使用Vue Router实现页面路由管理
- 使用Pinia实现全局状态管理
- 使用Axios与后端API交互
- 采用响应式设计，适配PC端和移动端

### 5.2 后端架构

- 采用Spring Boot分层架构：
  - Controller层：处理HTTP请求，返回响应
  - Service层：实现业务逻辑
  - DAO层：数据访问，与数据库交互
  - Model层：数据模型，对应数据库表结构
- 使用Spring Security实现Session认证
- 使用MyBatis-Plus简化数据库操作

### 5.3 前后端交互

- 前端通过Axios发送HTTP请求到后端API
- 后端返回JSON格式数据
- 前端解析数据并更新UI
- 使用RESTful API设计风格

## 6. 安全设计

### 6.1 认证与授权

- 使用Session认证，确保用户身份安全
- 基于角色的权限控制，防止越权操作
- 密码加密存储，使用BCrypt算法

### 6.2 数据安全

- 输入验证：对用户输入进行严格验证，防止SQL注入等攻击
- 数据传输：使用HTTPS协议，确保数据传输安全
- 敏感信息：对敏感信息进行加密处理

### 6.3 系统安全

- 定期备份数据库，防止数据丢失
- 限制登录尝试次数，防止暴力破解
- 日志记录，便于安全审计
